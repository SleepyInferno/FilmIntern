# Phase 3.1: Multi-Provider AI Support - Research

**Researched:** 2026-03-17
**Domain:** AI provider abstraction, settings persistence, Vercel AI SDK provider management
**Confidence:** HIGH

## Summary

This phase adds a global settings panel where users choose which AI provider (Anthropic Claude, OpenAI, or Ollama/local) powers the analysis pipeline. The existing codebase hardcodes `anthropic('claude-sonnet-4-5')` in a single API route (`src/app/api/analyze/route.ts`). The Vercel AI SDK (`ai` v6) already provides `createProviderRegistry` and first-class multi-provider support, making this a configuration/UI task rather than a deep refactor.

The key architectural decisions are: (1) use the AI SDK's `createProviderRegistry` to centralize provider resolution, (2) persist settings to a JSON file on disk (no database exists yet, this is a single-user personal tool), (3) expose a Settings page reachable from the existing sidebar, and (4) pass the selected provider/model to the analyze route at call time.

**Primary recommendation:** Use `createProviderRegistry` with `@ai-sdk/openai` and `ollama-ai-provider-v2` alongside the existing `@ai-sdk/anthropic`. Persist settings to a server-side JSON file. The sidebar already has a disabled Settings nav item -- enable it and link to a `/settings` page.

### Pattern 1: Provider Registry (Server-Side)
**What:** Centralized provider resolution using AI SDK's `createProviderRegistry`
**When to use:** When the analyze route needs to dynamically pick a provider/model
**Example:**
```typescript
// src/lib/ai/provider-registry.ts
// Source: https://ai-sdk.dev/docs/ai-sdk-core/provider-management
import { createProviderRegistry } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { openai } from '@ai-sdk/openai';
import { createOllama } from 'ollama-ai-provider-v2';

export function buildRegistry(ollamaBaseURL?: string) {
  return createProviderRegistry({
    anthropic,
    openai,
    ollama: createOllama({
      baseURL: ollamaBaseURL || 'http://localhost:11434/api',
    }),
  });
}

// Usage in route: registry.languageModel('anthropic:claude-sonnet-4-5')
```

### Pattern 2: Settings Persistence (JSON File)
**What:** Read/write a JSON settings file on disk for provider configuration
**When to use:** Single-user tool without a database
**Example:**
```typescript
// src/lib/ai/settings.ts
import { readFile, writeFile, mkdir } from 'fs/promises';
import path from 'path';

const SETTINGS_PATH = path.join(process.cwd(), '.filmintern', 'settings.json');

export interface AISettings {
  provider: 'anthropic' | 'openai' | 'ollama';
  anthropic: { model: string };          // e.g. 'claude-sonnet-4-5'
  openai: { model: string };             // e.g. 'gpt-4o'
  ollama: { model: string; baseURL: string };
}

export const DEFAULT_SETTINGS: AISettings = {
  provider: 'anthropic',
  anthropic: { model: 'claude-sonnet-4-5' },
  openai: { model: 'gpt-4o' },
  ollama: { model: 'llama3.1', baseURL: 'http://localhost:11434/api' },
};

export async function loadSettings(): Promise<AISettings> {
  try {
    const raw = await readFile(SETTINGS_PATH, 'utf-8');
    return { ...DEFAULT_SETTINGS, ...JSON.parse(raw) };
  } catch {
    return DEFAULT_SETTINGS;
  }
}

export async function saveSettings(settings: AISettings): Promise<void> {
  await mkdir(path.dirname(SETTINGS_PATH), { recursive: true });
  await writeFile(SETTINGS_PATH, JSON.stringify(settings, null, 2));
}
```

### Pattern 3: Dynamic Model Resolution in API Route
**What:** Read settings, resolve provider + model, pass to streamText
**When to use:** In the analyze route, replacing the hardcoded anthropic call
**Example:**
```typescript
// In src/app/api/analyze/route.ts
import { loadSettings } from '@/lib/ai/settings';
import { buildRegistry } from '@/lib/ai/provider-registry';

// Inside POST handler:
const settings = await loadSettings();
const registry = buildRegistry(settings.ollama.baseURL);
const modelId = {
  anthropic: `anthropic:${settings.anthropic.model}`,
  openai: `openai:${settings.openai.model}`,
  ollama: `ollama:${settings.ollama.model}`,
}[settings.provider];

const result = streamText({
  model: registry.languageModel(modelId),
  output: Output.object({ schema: config.schema }),
  system: config.prompt,
  prompt: `${inputTypePrefix}Analyze this material:\n\n${text}`,
  // Remove anthropic-specific providerOptions OR gate them
  ...(settings.provider === 'anthropic' ? {
    providerOptions: { anthropic: { structuredOutputMode: 'auto' } },
  } : {}),
  onError({ error }) {
    console.error('Analysis streaming error:', error);
  },
});
```

### Anti-Patterns to Avoid
- **Storing API keys in the settings JSON:** API keys should stay in `.env.local` environment variables. The settings file stores provider choice and model name only. The providers read their own env vars (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`) automatically.
- **Client-side provider instantiation:** Provider SDKs contain API keys and must only run server-side. The settings page sends config to a server API route.
- **Rebuilding registry on every request:** The registry is cheap to create but could be cached in a module-level variable, rebuilt only when settings change.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Provider abstraction | Custom if/else chain per provider | `createProviderRegistry` from `ai` | Handles model resolution, type safety, consistent API |
| OpenAI streaming | Custom fetch to OpenAI API | `@ai-sdk/openai` provider | Handles SSE parsing, token counting, error handling |
| Ollama integration | Custom fetch to Ollama HTTP API | `ollama-ai-provider-v2` | Handles structured output mode, model catalog, streaming |
| Settings form UI | Custom form components | shadcn/ui Select, Card, Button (already in project) | Consistent with existing UI |

## Common Pitfalls

### Pitfall 1: Structured Output Compatibility Across Providers
**What goes wrong:** The current route uses `Output.object({ schema })` with `providerOptions: { anthropic: { structuredOutputMode: 'auto' } }`. OpenAI supports structured output natively. Ollama support depends on the model.
**Why it happens:** Not all models/providers handle Zod schema-constrained output identically.
**How to avoid:** Keep `Output.object({ schema })` -- the AI SDK handles provider differences. Remove the `anthropic`-specific `providerOptions` or gate it to only apply for the Anthropic provider. For Ollama, warn users that smaller models may produce invalid JSON.
**Warning signs:** Analysis returning malformed/partial JSON with Ollama models.

### Pitfall 2: API Keys in Wrong Place
**What goes wrong:** Temptation to let users enter API keys in the settings UI and store them in the JSON file.
**Why it happens:** Seems user-friendly to configure everything in one place.
**How to avoid:** API keys stay in `.env.local`. The providers auto-read `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` from env. Ollama typically needs no key. The settings page can show which env vars are set (truthy/falsy) but never display or store the actual keys.
**Warning signs:** API keys visible in settings JSON file on disk.

### Pitfall 3: Sidebar Navigation State
**What goes wrong:** Adding a Settings page but not properly handling active state in the sidebar.
**Why it happens:** The sidebar currently hardcodes `active: true` on "Projects" and uses no routing.
**How to avoid:** The simplest approach: make the Settings button open a modal/dialog overlay rather than a full page route, avoiding routing complexity entirely. Alternatively, use `usePathname()` to determine active state if using a real route.
**Warning signs:** Both "Projects" and "Settings" appearing active simultaneously.

### Pitfall 4: Ollama Not Running
**What goes wrong:** User selects Ollama provider but Ollama is not running locally.
**Why it happens:** Ollama requires a separate local process (`ollama serve`).
**How to avoid:** Add a connection test endpoint or test the connection when saving Ollama settings. Show clear error message in the UI when Ollama is unreachable.
**Warning signs:** Hanging requests or cryptic ECONNREFUSED errors.

## Code Examples

### Settings API Route
```typescript
// src/app/api/settings/route.ts
import { loadSettings, saveSettings, AISettings } from '@/lib/ai/settings';

export async function GET() {
  const settings = await loadSettings();
  return Response.json(settings);
}

export async function PUT(req: Request) {
  const body = await req.json() as AISettings;
  // Validate shape before saving
  await saveSettings(body);
  return Response.json({ ok: true });
}
```

## Open Questions

1. **Settings UI: Page vs Dialog?**
   - What we know: Sidebar has a disabled Settings button. Adding a `/settings` route requires routing logic.
   - What's unclear: Whether a dialog/modal would be simpler for a few settings fields.
   - Recommendation: Use a `/settings` page route. It is cleaner for future expansion (export settings, history settings in v2). Enable the sidebar Settings button as a link.

2. **Model Selection: Free-text vs Dropdown?**
   - What we know: OpenAI and Anthropic have known model lists. Ollama models are user-installed and vary.
   - What's unclear: Whether to enumerate models or let user type.
   - Recommendation: Use text input for model name with sensible defaults pre-filled. Avoids maintaining model lists that go stale.

3. **Env Var Display**
   - What we know: Users need to set ANTHROPIC_API_KEY and OPENAI_API_KEY in .env.local
   - What's unclear: How much guidance to show in the settings UI
   - Recommendation: Show a status indicator (green/red) for whether each provider's API key env var is detected. Do not display the key value.
